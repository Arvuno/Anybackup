# AG-UI 内容规则

## 协议化内容约束

Markdown 设计约束必须让后续 `ag-ui-response` 能生成明确的 AG-UI draft，而不是只给文案方向。每次设计至少说明：

- 输出意图：对应 `meta.intent` 的 `thought`、`tool_call`、`clarification`、`result` 或 `error`。
- 卡片内容：标题、摘要、事实字段、风险提示、用户动作和状态变化。
- 组件选择：每一类内容使用哪些 layout node，例如 `card`、`kv-list`、`data-table`、`callout`、`metric-list`、`badge-row`、`action-row`。
- 动作定义：哪些按钮需要进入 `actions`，按钮文案、主次关系、触发后的业务含义。
- 状态定义：是否需要 `state_snapshot` 或 `state_delta`，以及 `interaction.status`、选择状态和视图状态如何变化。
- 更新方式：新建卡片使用完整 activity，更新进度或状态使用 patch，不把持续变化字段埋在长段落里。

## 思考链摘要

- 只展示用户可见的阶段判断，不展示内部推理链。
- 内容包含：阶段名称、当前已知事实、下一步动作、关键风险或等待事项。
- 可用阶段：意图理解、信息获取、策略规划、风险确认、下发执行、执行观察、可用性验证、最终反馈。

## 工具调用展示

- 工具调用必须转换成业务语义能力名称，例如“恢复知识检索”“恢复对象查询”“恢复点查询”“恢复可用性验证”。
- 展示字段包含：业务能力、状态、调用目的、脱敏输入摘要、脱敏结果摘要。
- 输入摘要只描述范围、对象类别、时间范围和筛选条件，不展示原始参数。
- 结果摘要只展示命中数量、业务结论、空结果依据或下一步，不展示原始返回。

## 主动询问

- 使用轻量对话卡片说明缺失信息、为什么必须确认、用户可选择的选项、默认建议和风险。
- 问题必须具体，例如“请告诉我是哪一个系统的数据”“什么时间做备份对业务影响最小”“如果发生故障，最多能接受丢失多久的数据”。
- 选项按钮组必须和问题一一对应；推荐选项放第一个或使用 primary 样式。
- 动作使用“确认继续”“补充条件”“调整范围”“取消任务”等业务语义。

## 结果返回

- 终态 `result`、最终报告和终态 `error` 的设计必须在 AI 已完成结果输出后进行；把该输出作为结果源文本，并在发布前冻结。
- 设计约束只能从结果源文本中抽取标题、摘要、事实、风险、动作和状态，不得新增结果源文本之外的事实、结论、时间点、数量、方案或风险等级。
- 如果卡片需要补充结果源文本中没有的关键字段，先修订 AI 结果输出，再重新设计 AG-UI；不得让 AG-UI 卡片和 AI 结果各自独立生成。
- 设计约束必须要求后续 `ag-ui-response` 做一致性校验：卡片标题、核心结论、事实字段、风险提示和动作必须与结果源文本一致。
- 不得提前发布终态 `result`；执行中进展、工具状态和阶段性发现可以用 `thought`、`tool_call` 或阶段性 `result` 表达，但最终结果必须等待结果源文本稳定。
- 标题必须表达业务结论，例如“已找到最新可用恢复点”“未找到可用恢复点”“需要确认恢复范围”。
- 摘要先给结论，再给证据字段、检查范围、候选方案、风险和下一步。
- 多候选方案展示为并列卡片；每个候选展示时间点、对象、可用性、风险、推荐理由和动作。
- 空结果必须展示已检查范围、未命中原因、可补充条件或重试动作。

## 备份方案审查

- 使用方案审查卡，不用普通结果段落。
- 标题为“对象名 + 备份方案”或“对象名 + 变更方案”，副标题展示对象类型、数据库版本、数据量和业务系统。
- 核心参数使用事实表格：备份类型、策略、RPO/RTO、加密、存储目标或保留策略。
- AI 生成思路展示为短步骤：识别系统、匹配用户要求、推荐策略、冲突检查、结论建议。
- 展示信心条和百分比；低于阈值时必须给出“调整参数”而不是直接审批。
- 固定操作栏必须包含返回、调整参数、审核通过或同类动作。

## 运维和告警

- 运营概览卡先展示全局指标，再展示需要关注的问题表格，最后给出可选择的问题详情动作。
- 告警详情卡先展示优先级、系统、数据量和故障现象，再展示失败记录、根因分析和建议方案。
- 根因和建议必须用可扫描的事实表格或方案表，不写成长段落。
- 方案对比卡用于两个以上处理方案，推荐方案必须显示推荐理由、信心值、预计效果和风险等级。

## 恢复演练和校验

- 演练计划使用轻量对话卡片或方案卡，展示验证目标、最近备份、备份大小、预计耗时和演练环境。
- 演练结果卡必须展示恢复数据量、实际 RTO、备份时间点、RPO 达成情况。
- 数据完整性校验必须使用 `data-table` 展示校验项、源库、恢复后和结果。

## 错误返回

- 用用户可理解的脱敏原因描述，不展示异常类型、堆栈、内部地址或原始响应。
- 必须包含影响范围、当前状态、可重试性和下一步处理。

## 最终报告

- 展示执行结果、关键时间点、命中对象、验证状态、风险、后续动作和受控附件引用。
- 报告应使用 `metric-list`、`data-table`、`chart` 或 `attachment-list`，不使用长段落堆叠。
