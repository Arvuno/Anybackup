# AG-UI 布局模式

## 通用布局

- `stack`：单阶段线性信息，适合 thought、tool_call、result、error 的根节点。
- `grid`：多个候选方案或对比项，子节点优先使用 `card`。
- `card`：一个候选方案、一个恢复点或一个执行阶段的独立摘要。
- `kv-list`：事实字段、状态、脱敏输入摘要、结果摘要、风险摘要。
- `data-table`：同构清单，例如恢复点列表、检查项、候选方案明细、空结果依据。
- `callout`：风险、限制、空结果、失败原因或关键提示。
- `action-row`：确认、重试、查看详情、补充条件、选择候选方案。
- `metric-list`、`chart`、`attachment-list`：报告、趋势、指标、可控附件引用。

## AG-UI layout-tree 设计方式

布局设计必须贴合 `conversation.ui.layout-tree@1`，先定义根节点和子节点职责，再描述卡片内容，而不是只描述视觉效果。

- 根布局：单个输出默认 `stack`；需要并列比较时使用 `grid`；需要把一个方案作为独立审查对象时使用 `card`。
- 节点顺序：`heading` 或标题型 `paragraph` 在前，事实节点居中，风险和依据靠后，`action-row` 放在末尾。
- 结构化优先：只要出现业务字段、候选对象、检查结果、风险或状态，必须使用 `kv-list`、`data-table`、`metric-list`、`badge-row` 或 `callout`。
- 更新友好：会持续变化的进度、状态、命中数量和风险结论放在独立节点，便于后续 `ACTIVITY_DELTA` patch。
- 状态联动：卡片内容涉及等待用户、执行中、已完成或失败时，同步要求 `state_snapshot` 或 `state_delta`。

## 卡片内容模型

设计卡片时按以下结构取舍，避免把所有内容塞入段落：

| 区域 | 推荐节点 | 设计重点 |
| --- | --- | --- |
| 标题区 | `heading`、`badge-row` | 业务对象、结论、状态、推荐或待审核标记 |
| 摘要区 | `paragraph`、`callout` | 一到两句说明当前结论或需要用户完成的动作 |
| 事实区 | `kv-list`、`data-table` | 系统、数据库、地址、数据量、时间点、RPO/RTO、策略 |
| 依据区 | `callout`、详情折叠区 | 用户可验证的脱敏证据、AI 生成思路摘要、冲突说明 |
| 风险区 | `callout`、`badge-row` | 失败、风险等级、影响范围、限制条件 |
| 动作区 | `action-row` | 主动作唯一，次动作克制，动作文案使用业务语义 |

## 组件选择矩阵

| 决策目标 | 推荐组件组合 | 说明 |
| --- | --- | --- |
| 让用户选择对象 | 轻量对话卡片 + `data-table` 或短列表 + `action-row` | 对象少时按钮直接显示业务代号；对象多时先表格再动作 |
| 让用户补充信息 | `callout` + `kv-list` + `action-row` | 说明缺失项、为什么需要、可选答案和默认建议 |
| 展示工具执行进度 | `heading` + `kv-list` + `badge-row` 或 `metric-list` | 展示业务能力、状态、范围、阶段进度和下一步 |
| 展示普通结果 | `callout` + `kv-list`，可加 `data-table` | 结论先行，事实支撑，最后给下一步 |
| 展示候选方案 | `grid` + 多个 `card` + 每卡 `action-row` | 每个方案独立展示指标、风险、推荐理由和动作 |
| 展示错误或空结果 | `callout` + `kv-list` + 可选 `action-row` | 说明检查范围、脱敏原因、影响和重试或补充动作 |
| 展示审查方案 | `card` + 参数网格 + 信心条 + 详情折叠区 + 固定操作栏 | 适合备份方案、变更方案、恢复演练方案 |

## 对话式企业运维风格基线

- 轻量对话卡片：白底圆角、轻边框、轻阴影，正文克制；首句确认用户意图，随后用加粗问题引导用户选择或确认。
- 思考条：作为独立的灰底可折叠区域，仅放简短阶段摘要和完成状态，不承载主要业务内容。
- 事实表格：优先用 `kv-list` 或紧凑 `data-table` 表达两列事实；行间距小、字段名短、值突出，适合系统、地址、数据量、RPO/RTO、备份窗口。
- 选项按钮组：使用 `action-row` 紧贴卡片下方；主推荐按钮使用 primary，其余使用 secondary；按钮文案短于 12 个汉字或等宽业务代号。
- 状态圆点和徽标：用 `badge-row` 或表格状态列表达成功、告警、待审核、风险级别和推荐，不在正文中反复解释颜色。
- 详情折叠区：长说明、AI 依据、保留策略、存储目标、风险和执行日志放入可展开详情；默认先展示结论和关键参数。
- 固定操作栏：审查、审批、调整参数、返回等主动作放在卡片底部；主按钮放右侧，次按钮放左侧或中间。

## 场景布局

| 场景 | 根布局 | 必备组件 | 说明 |
| --- | --- | --- | --- |
| `thought` | `stack` | `heading`、`paragraph` | 只展示可见阶段摘要和下一步。极短进度可只用简短结构。 |
| `tool_call` | `stack` | `heading`、`kv-list` | 必须展示业务语义能力名称、状态、目的、脱敏输入摘要。 |
| 工具完成 | `stack` | `kv-list`、`data-table` 或 `callout` | 展示脱敏结果、命中数量、空结果依据或失败摘要。 |
| `clarification` | `stack` | `callout`、`kv-list`、`action-row` | 说明缺失信息、风险确认项和等待用户选择的原因。 |
| 候选方案 | `stack` + `grid` | `card`、`metric-list`、`kv-list`、`action-row` | 多方案并列比较，并提供确认或修订动作。 |
| 正常结果 | `stack` | `callout` 或 `kv-list`，可加 `data-table` | 结论先行，再给证据、字段和下一步。 |
| 空结果 | `stack` | `callout`、`kv-list`、`data-table` | 说明未命中的检查范围、原因和可执行下一步。 |
| 错误结果 | `stack` | `callout`、`kv-list`、可选 `action-row` | 展示脱敏原因、影响范围、下一步处理。 |
| 最终报告 | `stack` | `metric-list`、`data-table`，可加 `chart`、`attachment-list` | 适合展示指标、明细、趋势和受控报告引用。 |

## 业务卡片模式

| 模式 | 适用场景 | 结构 |
| --- | --- | --- |
| 轻量对话卡片 | 选择系统、确认时间、询问数据库类型、确认优先级 | `stack` + 短 `paragraph` + 加粗问题 + `data-table` 或 bullet list + `action-row` |
| 迁移检测卡 | 检查迁移后配置、旧地址是否继续使用、连通性验证 | `stack` + 标题 + 事实表格 + 配置状态列表 + 风险 `callout` + 检测结果 + 追问 + `action-row` |
| 配置变更卡 | 生成备份目标切换、旧配置下线、策略保持不变等方案 | `stack` + 标题 + 事实表格 + 检查列表 + 成功 `callout` + 链接式 `action-row` |
| 方案审查卡 | 备份方案、容灾方案、配置变更方案需要管理员审核 | `card` + 标题行 + 信心徽标 + 待审核徽标 + 2x2 参数网格 + AI 生成思路 + 信心条 + 详情折叠区 + 固定操作栏 |
| 运营概览卡 | 全局运行状态、成功率、容量、告警数量 | `stack` + `kv-list` 指标概览 + 告警 `data-table` + 摘要指标卡 + `action-row` |
| 告警详情卡 | P1/P2 告警、失败记录、根因、建议方案 | `stack` + 告警标题 + 事实表格 + 失败记录列表 + 根因 `callout` + 建议方案表格 |
| 方案对比卡 | 需要在多个修复、备份或恢复方案中选择 | `grid` + 多个 `card` + 推荐徽标 + 信心值 + 关键指标 + 详情折叠区 + 每卡独立动作 |
| 演练结果卡 | 恢复演练、快速校验、数据完整性校验完成 | `stack` + 结果标题 + `kv-list` + 校验 `data-table` + 结论 `callout` |

## 方案审查卡细节

- 标题行包含保护对象或方案名、保护类型图标、信心百分比、待审核或已通过状态。
- 参数区使用 2x2 的事实表格：备份类型、策略、RPO/RTO、加密或存储目标。
- AI 生成思路必须放在浅蓝信息区，使用“识别、需求、推荐、冲突、建议”这样的短链路。
- 信心条使用 `metric-list` 或进度型 `chart` 表达百分比，文字值与条形值一致。
- 详情折叠区默认收起；展开后展示保留策略、存储目标、AI 分析、风险和工作动态。
- 固定操作栏提供“返回”“调整参数”“审核通过”或同类动作，主动作只有一个。

## 组件选择边界

- 业务结果、工具调用、错误和候选方案不得只设计为段落。
- 有列表就优先设计 `data-table` 或 `grid`，不要压缩成自然语言长段。
- 有用户动作就定义 `actions`，并通过 `action-row` 展示。
- 有状态变化就要求后续 draft 携带 `state_snapshot` 或 `state_delta`。
