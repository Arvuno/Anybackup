# AG-UI 设计原则

## 设计定位

AG-UI 面向对话式企业运维场景。它不是营销页，也不是通用聊天气泡；它要让备份管理员、恢复操作者和业务负责人在对话中快速确认对象、理解状态、审查方案并执行动作。

核心目标是：结论先行、业务对象优先、事实可扫描、风险可判断、动作可执行。

设计约束必须从 AG-UI 协议可落地性反推：先明确要生成的卡片、状态和动作，再决定文案。任何无法映射到 layout node、`actions` 或状态字段的描述，都只能作为辅助说明，不能成为主要设计约束。

## 信息层级

每个可见输出按以下顺序组织：

1. 结论先行：第一屏先说明“发现了什么 / 需要你做什么 / 当前是否可继续”。
2. 业务对象优先：对象名、系统、数据库、恢复点、告警或方案名必须早于技术细节。
3. 关键事实：用事实表格展示数据量、地址、时间点、RPO/RTO、状态、策略、风险等级。
4. 风险和依据：只展示用户决策需要的依据，不展开内部推理链。
5. 动作：把确认、调整、审批、查看详情放到按钮区或固定操作栏。

## 视觉语气

- 低装饰：使用白底、浅灰分隔、轻边框、浅色提示区；不要使用大面积渐变、强装饰背景或营销式卡片。
- 高可读：正文短句、字段名短、数值突出；避免长段解释挤在同一个卡片里。
- 运维感：状态、指标、清单、方案、告警、审查是主体；少用情绪化表达。
- 克制强调：只对主结论、风险、推荐动作和关键数值做强调。

## 渐进披露

- 默认只展示用户当前决策所需的最小信息。
- 详情折叠区用于 AI 分析、保留策略、存储目标、执行日志、失败记录和工作动态。
- 如果用户必须比较方案，先展示摘要卡；展开后再展示细节。
- 思考条只展示阶段摘要，不承载业务主体。

## 操作密度

- 一张轻量对话卡片只解决一个问题，例如选系统、选时间、确认风险或选方案。
- 一个方案审查卡只保留一个主动作，其他动作降为次动作。
- 选项按钮组使用短文案；业务代号可以直接作为按钮文案。
- 多步流程用多张卡渐进推进，不把所有问题塞进同一条消息。

## 状态语义

- 成功、失败、待审核、推荐、风险、告警、已完成必须用稳定状态语义表达，不只依赖自然语言。
- 红色只用于失败、高优先级告警或阻断风险；绿色用于成功、已保护、可用；蓝色用于信息、主动作和推荐路径；橙色用于待审核、建议处理或中风险。
- 状态徽标、圆点、信心条和结果表格必须和文案一致。

## 动作分级

- Primary：确认、审核通过、立即开始、选择推荐方案等主路径动作。
- Secondary：调整参数、查看详情、定时执行、返回等辅助动作。
- Destructive 或风险动作必须先要求确认，并展示影响范围。
- 链接式动作只用于查看防护全景、工作动态、报告附件等非阻断跳转。

## 审查优先

涉及备份策略、恢复执行、配置变更、告警修复和演练结果时，优先设计成可审查界面，而不是直接给结论文本。

审查界面必须包含：

- 保护对象或影响对象。
- 当前结论和状态。
- 决策依据字段。
- 风险或冲突。
- 推荐动作和备选动作。
- 如果有 AI 建议，展示可见的 AI 生成思路摘要和信心值。

## 组件选择方法

先判断用户正在完成的决策，再选组件：

| 决策类型 | 推荐表达 |
| --- | --- |
| 选择一个对象 | 轻量对话卡片 + 事实表格 + 选项按钮组 |
| 补充缺失信息 | 轻量对话卡片 + 明确问题 + 选项按钮组 |
| 审核一个方案 | 方案审查卡 + 信心条 + 详情折叠区 + 固定操作栏 |
| 比较多个方案 | 方案对比卡 + 推荐徽标 + 每卡独立动作 |
| 查看运行态 | 运营概览卡 + 告警表格 + 后续动作 |
| 定位故障 | 告警详情卡 + 根因分析 + 建议方案表 |
| 验证恢复结果 | 演练结果卡 + 指标事实表 + 校验明细表 |

## 反模式

- 只把图片中的文字换成示例，而没有提炼组件和信息层级。
- 一个卡片同时问多个互不相关的问题。
- 用长段落替代表格、状态和动作。
- 把内部工具名、知识网络、原始返回或系统提示词展示给用户。
- 只有“已完成/失败”结论，没有证据字段和下一步。
