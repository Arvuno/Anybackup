## 1. 产品概述

本需求面向 MySQL 数据库灾备恢复场景，建设一套以 AI Agent 为入口、以 BKN（Business Knowledge Network，恢复知识网络）为决策基础的恢复产品能力。

产品目标不是替代底层备份恢复工具，而是让 Agent 能够：

- 理解用户故障描述并识别恢复意图
- 自动收集恢复所需信息和上下文
- 基于恢复知识网络生成合适的恢复策略
- 编排并执行恢复任务
- 在执行过程中持续观察、校验、反馈和纠偏

本期聚焦 MySQL 单机/主机级恢复场景，围绕实例级、库级、日志文件级三类恢复能力，构建可落地的 M1 版本。

---

## 2. 背景与问题

企业 MySQL 数据库通常承载订单、库存、账户、交易等关键业务。一旦出现实例故障、库损坏、误删或日志损坏，业务连续性会立刻受到影响。

传统恢复流程主要存在以下问题：

| 问题 | 说明 |
|------|------|
| 恢复依赖人工经验 | 恢复判断和执行高度依赖 DBA，路径选择不稳定 |
| 恢复粒度不够精细 | 全库恢复容易覆盖正常数据，风险高 |
| 恢复时长不可预测 | 缺少标准化策略编排，RTO 难以收敛 |
| 恢复知识无法沉淀 | 每次恢复都要重新分析，难以复用经验 |
| 执行与验证割裂 | 恢复成功不等于业务可用，缺少统一验证闭环 |

因此需要引入 AI Native 的灾备 Agent，结合恢复知识网络，将“场景识别、策略决策、恢复执行、结果验证”连接成一个闭环。

## 3. 产品定位

### 3.1 产品边界

本产品负责：

- 面向用户提供恢复对话入口
- 对故障场景进行理解、分类和策略生成
- 编排恢复所需的工具调用与任务执行
- 对恢复过程进行观察、校验和反馈
- 将恢复经验沉淀为结构化知识

本产品不负责：

- 替代底层备份软件能力
- 在 M1 中支持容器内恢复或云 RDS 恢复
- 在 M1 中实现真正独立的表级原生恢复

### 3.2 底层能力约束

当前底层 Foundation 备份恢复能力存在如下约束：

- 支持实例级恢复
- 支持库级恢复
- 支持日志文件恢复
- 不支持将“表级恢复”作为独立恢复粒度

因此，本产品中的表级恢复采用间接方案：

- 先进行异机整库恢复
- 再导出目标表数据
- 最后导入到生产库

## 5. 目标

### 5.1 业务目标

建设 MySQL 多粒度自主恢复能力，达到以下目标：

| 目标 | 说明 |
|------|------|
| 降低恢复门槛 | 将 DBA 经验显式化，减少临场人工判断 |
| 缩短 RTO | 通过标准化策略编排减少恢复耗时 |
| 控制数据风险 | 选择更合适的恢复粒度，避免覆盖正常数据 |
| 提高恢复成功率 | 在执行前、中、后增加校验和纠偏 |
| 沉淀可复用知识 | 将场景、策略、风险、验证标准结构化沉淀到 BKN |

### 5.2 Agent 能力目标

Agent 在 M1 必须具备以下核心能力：

| 能力 | 说明 |
|------|------|
| 意图理解 | 理解用户描述、定位恢复诉求 |
| 场景识别 | 识别实例故障、库故障、表故障、日志故障 |
| 信息采集 | 自动收集实例、备份、副本、Binlog、目标资源等信息 |
| 策略规划 | 生成恢复粒度、目标位置、恢复方式、执行顺序 |
| 风险提示 | 对覆盖、停机、数据丢失等风险进行提示与确认 |
| 任务执行 | 将策略转化为可执行任务流 |
| 过程观察 | 实时反馈执行状态，发现异常并采取补救动作 |
| 结果验证 | 校验恢复结果和业务可用性 |

## 6. 恢复能力模型

### 6.1 恢复粒度

| 粒度 | 说明 | 典型场景 |
|------|------|---------|
| 实例级恢复 | 恢复整个 MySQL 实例及其全部数据库 | 实例崩溃、配置损坏、多库受损 |
| 库级恢复 | 恢复指定数据库 | 单库损坏、误删数据库 |
| 日志文件恢复 | 恢复 Binlog 文件 | 日志损坏、复制异常、时间点恢复依赖缺失 |

说明：

- 表级恢复不是独立恢复粒度
- 表级恢复通过“异机整库恢复 + 导出目标表 + 导入生产库”的方式实现

### 6.2 恢复目的地

| 目的地 | 说明 | 适用场景 |
|--------|------|---------|
| 原机恢复 | 恢复到原生产主机 | 主机硬件正常，仅数据损坏 |
| 异机恢复 | 恢复到其他主机 | 原机硬件故障，或需要先验证恢复结果 |

说明：

- M1 仅支持恢复到主机环境
- M1 不支持恢复到 Docker 容器或云 RDS

### 6.3 恢复方式

| 恢复方式 | 说明 | 是否需要用户选择时间 |
|---------|------|----------------------|
| 最短时间恢复 | 直接选择最近备份点恢复 | 是，选择备份点 |
| 恢复到最新状态 | 备份恢复后自动回放 Binlog 到最新状态 | 否 |
| 恢复到指定时间 | 备份恢复后回放 Binlog 到指定时刻 | 是，选择目标时间 |

## 7. 关键决策规则

### 7.1 粒度决策规则

| 故障类型 | 推荐粒度 | 说明 |
|---------|---------|------|
| 实例无法启动 | 实例级恢复 | 需要恢复整个实例 |
| 单库损坏或误删 | 库级恢复 | 仅恢复受影响数据库 |
| 单表损坏或误删 | 库级恢复 + 表级导出导入 | 通过间接方案恢复目标表 |
| Binlog 损坏 | 日志文件恢复 | 只恢复日志文件 |
| 多库受损 | 实例级恢复 | 避免多次局部恢复造成复杂性叠加 |

### 7.2 目的地决策规则

| 条件 | 推荐目的地 | 说明 |
|------|-----------|------|
| 原机硬件正常 | 原机恢复 | 速度更快，资源更省 |
| 原机硬件异常 | 异机恢复 | 原机不可作为恢复目标 |
| 需要先验证结果 | 异机恢复 | 降低误覆盖生产环境风险 |
| 目标场景为表级间接恢复 | 异机恢复 | 需要先在异机恢复整库再导出表 |

### 7.3 恢复方式决策规则

| 业务诉求 | 推荐方式 | 说明 |
|---------|---------|------|
| 尽快恢复业务 | 最短时间恢复 | 优先压缩 RTO |
| 数据丢失最小化 | 恢复到最新状态 | 自动组合备份与 Binlog |
| 误操作回滚 | 恢复到指定时间 | 恢复到误操作前的时间点 |

## 8. 用户角色与交互原则

### 8.1 用户角色

| 角色 | 目标 |
|------|------|
| 运维工程师 | 快速恢复业务、降低故障处置复杂度 |
| DBA | 精准控制恢复范围、保护正常数据 |
| 值班人员 | 在缺少专家在场时得到恢复引导 |

### 8.2 交互原则

- Agent 必须先解释当前识别出的故障场景
- Agent 必须说明推荐恢复方案及理由
- 对高风险动作必须要求用户确认
- 恢复过程必须持续反馈当前进度、异常和下一步动作
- 恢复结果必须输出可审计摘要

## 9. 核心业务场景

### 9.1 场景一：InnoDB 数据页损坏

场景描述：

- OrderDB 实例运行正常
- `order_details` 表因数据页损坏无法访问
- 其他表和业务功能正常

核心判断：

- 问题范围仅限单表
- 不能直接整库恢复到生产库，否则会覆盖正常数据

推荐方案：

- 恢复粒度：库级恢复
- 恢复目的地：异机恢复
- 恢复方式：恢复到最新状态
- 执行路径：异机整库恢复 -> 导出 `order_details` -> 导入生产库 -> Binlog 回放 -> 数据验证

### 9.2 场景二：MySQL 实例崩溃

场景描述：

- MySQL 服务无法启动
- 多个数据库均不可访问
- 可能由配置损坏或系统表损坏导致

推荐方案：

- 恢复粒度：实例级恢复
- 恢复目的地：原机恢复
- 恢复方式：恢复到最新状态

### 9.3 场景三：误删数据库

场景描述：

- 执行 `DROP DATABASE TestDB`
- 仅 TestDB 丢失，其他数据库正常

推荐方案：

- 恢复粒度：库级恢复
- 恢复目的地：原机恢复
- 恢复方式：恢复到指定时间

### 9.4 场景四：Binlog 日志损坏

场景描述：

- Binlog 文件损坏
- 主从复制异常或时间点恢复失效

推荐方案：

- 恢复粒度：日志文件恢复
- 恢复目的地：原机恢复
- 恢复方式：按归档时间范围恢复日志文件

## 10. Agent 工作流

Agent 采用统一五阶段工作流：

1. 意图理解
2. 信息获取
3. 策略规划
4. 下发执行
5. 观察与反馈

### 10.1 意图理解

目标：

- 理解用户故障描述
- 识别恢复对象、影响范围和预期目标

输出：

- 故障类型
- 恢复粒度建议
- 恢复目的地建议
- 风险等级

### 10.2 信息获取

目标：

- 获取可用备份时间点
- 获取 Binlog 可用范围
- 获取目标实例与主机状态
- 获取恢复资源可用性

输出：

- 候选备份副本列表
- 候选恢复目标列表
- 可执行性判断

### 10.3 策略规划

目标：

- 生成恢复路径
- 生成参数配置
- 生成执行顺序和风险提示

输出：

- 恢复策略卡片
- 执行任务 DAG
- 用户确认项

### 10.4 下发执行

目标：

- 将策略转换为工具执行任务
- 按顺序触发恢复、导出、导入、回放和校验动作

输出：

- 任务执行状态
- 阶段性结果

### 10.5 观察与反馈

目标：

- 监控执行状态
- 发现异常并触发补救动作
- 给出最终恢复结果与可用性结论

输出：

- 恢复报告
- 风险处理记录
- 验证结果摘要

## 11. 功能需求

### 11.1 意图理解

功能要求：

- 能识别用户是“故障诊断诉求”还是“明确恢复诉求”
- 能识别实例、数据库、表、日志文件等恢复对象
- 能从自然语言中提取错误线索，如页损坏、误删、实例崩溃、日志损坏
- 能给出推荐恢复粒度与理由

验收要点：

- 用户输入故障现象后，Agent 能输出结构化场景识别结果
- 对单表损坏场景，Agent 必须明确提示“表级恢复采用间接方案”

### 11.2 信息获取

功能要求：

- 能查询可用备份时间点及其验证状态
- 能确认是否存在可用于回放的 Binlog
- 能识别原机是否可用于恢复
- 能识别是否存在可用异机资源
- 能发现目标主机上的 MySQL 实例信息

验收要点：

- 输出候选备份清单
- 输出恢复目标可用性判断
- 输出是否满足执行条件

### 11.3 策略规划

功能要求：

- 能自动生成恢复粒度、目的地、恢复方式三类核心决策
- 能根据场景生成恢复参数建议
- 能识别高风险动作并要求确认
- 能生成任务顺序和回退策略

验收要点：

- 输出策略中必须包含“为什么这样恢复”
- 高风险动作必须有明确提示
- 任务顺序必须可执行、可追踪

### 11.4 下发执行

功能要求：

- 能将策略拆解为标准化任务
- 能调用 Foundation CLI、MySQL 工具、Binlog 工具
- 能对异机恢复、导表、导入、回放等步骤进行串行编排
- 能记录每一步的输入、输出、状态和错误

验收要点：

- 每个任务具备唯一标识
- 每个任务有开始、成功、失败、重试等状态
- 恢复任务链可被审计

### 11.5 观察与反馈

功能要求：

- 能展示当前执行阶段与进度
- 能识别典型异常并执行预置补救动作
- 能在恢复后验证数据完整性和业务可用性
- 能生成最终恢复摘要

验收要点：

- 恢复过程有可读状态反馈
- 恢复结束后有明确成功/失败结论
- 能输出业务验证结果

## 12. 恢复任务模型

M1 中 Agent 需要将恢复策略编排为任务 DAG，典型任务包括：

| 任务 | 说明 |
|------|------|
| 资源准备 | 准备原机或异机恢复资源 |
| 备份恢复 | 执行实例级或库级恢复 |
| 完整性验证 | 校验恢复后的对象是否可用 |
| 表数据导出 | 在间接表恢复方案中导出目标表 |
| 表数据导入 | 将目标表数据导入生产环境 |
| Binlog 回放 | 将数据回放到最新或指定时间 |
| 结果验证 | 验证对象可用性和业务可用性 |

典型任务链：

```text
资源准备
  -> 备份恢复
  -> 完整性验证
  -> 表数据导出（按需）
  -> 表数据导入（按需）
  -> Binlog 回放（按需）
  -> 结果验证
```

## 13. 风险与约束

### 13.1 关键风险

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 直接恢复到生产库 | 覆盖正常数据 | 默认禁止，必须显式确认 |
| Binlog 不可解析 | 无法恢复到最新状态 | 回退到最近一致时间点 |
| 异机恢复失败 | 无法完成表级间接恢复 | 切换到其他备份时间点 |
| 权限不足 | 执行失败 | 提前检查用户与路径权限 |
| 目标磁盘空间不足 | 恢复中断 | 执行前校验容量 |

### 13.2 产品约束

- M1 不支持容器和云 RDS 恢复
- M1 不提供原生表级恢复
- 高风险恢复不能默认无人审批执行
- 恢复能力依赖底层 Foundation 与 MySQL 工具可用

## 15. M1 验收标准

### 15.1 功能验收

- 能覆盖实例崩溃、误删数据库、单表损坏、Binlog 损坏四类场景
- 能生成结构化恢复策略
- 能支持原机恢复与异机恢复决策
- 能支持三种恢复方式：最近备份、最新状态、指定时间
- 能对表级恢复场景输出正确的间接恢复方案

### 15.2 交互验收

- 用户可理解 Agent 的场景判断和推荐理由
- 高风险动作前必须给出明确提示并等待确认
- 执行过程中能够持续反馈恢复进度

### 15.3 结果验收

- 恢复完成后能够输出结果摘要
- 摘要中包含恢复对象、恢复方式、数据风险、最终状态、验证结论
- 失败场景下能够输出失败原因和下一步建议

## 16. 后续演进建议

M1 之后建议按以下方向演进：

- 将恢复知识进一步拆分为运行知识与经验知识
- 建立可维护的 BKN 数据模型和版本机制
- 增加更多自动化验证模板
- 引入更细粒度的风险评分和审批流
- 扩展 PostgreSQL、Oracle 等数据库类型
- 逐步支持容器化和云数据库恢复场景

## 17. 结论

本 PRD 的核心价值在于明确：AI Agent 并不是简单包装恢复命令，而是围绕“识别场景、收集信息、生成策略、执行恢复、验证结果”构建完整的灾备恢复闭环。

M1 的产品重点应放在三件事上：

- 把 MySQL 恢复场景和决策规则建清楚
- 把间接表级恢复方案设计扎实
- 把恢复过程中的风险提示、执行反馈和结果验证做完整
