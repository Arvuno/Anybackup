# 备份方案知识库设计文档

## 1. 知识库概述

备份方案知识库主要由两个核心知识网络组成，旨在支撑 AnyBackup 系统的智能化“备份方案推荐”与“备份运行配置关联”两大业务场景：

* **备份规则知识网络（backup_rule_knowledge_network）**：面向“备份方案推荐”场景的新知识网络。该网络以 `backup_solution`（备份方案）作为推荐结果的中心对象，通过剥离硬编码，将多维度的业务规则（应用、数据、法规、RPO等）作为独立的约束来源，动态汇总基础配置与应用配置，并最终通过灵活的输出规则进行结构化组装，实现智能化、可解释的方案推荐与决策。
* **备份运行知识网络（backup_run_knowledge_network）**：面向“备份环境运行态”，提供运行事实的数据支撑。用于精确描述保护对象、生产系统资源以及各项备份配置（如客户端、存储池、区域配置等）之间的物理和逻辑映射关系，提供系统运行状态数据的查询动作（Action）能力，为排障分析、环境巡检和配置核对提供真实、客观的环境事实。

## 2. 知识库组成

### 2.1 备份规则知识网络组成
围绕“备份方案推荐”的核心概念构建，主要由以下几类对象（Object Types）与关系（Relation Types）构成：

* **核心方案对象**：
  * `backup_solution`（备份方案）：推荐的中心结果对象。
  * `backup_window`（备份窗口）、`backup_plan`（备份计划）、`retention_plan`（保留计划）：作为组件对象，统一归集到具体的备份方案下。
* **约束规则对象**：
  * `app_rule`（应用级规则）、`data_rule`（数据规则）、`legal_rule`（法规规则）、`rpo_rule`（RPO规则）：分别从应用类型、数据分级属性、合规要求和目标RPO维度对候选方案进行严格的“缩圈”约束。
* **配置规则对象**：
  * `basic_config_rule`（基础配置）与 `app_config_rule`（应用配置）：基于应用类型、数据量等条件动态匹配的配置规则项集合，推荐阶段按配置类别进行动态汇总。
* **策略与输出控制对象**：
  * `recommendation_policy`（候选推荐策略）：提供行业级默认值、法规缩圈边界、评分权重及最大候选数量等。
  * `app_output_rule`（应用输出规则）：定义不同应用类型的最终展示结构、节点命名和字段隐藏策略，彻底避免在底层硬编码展示层级。
* **核心关系 (Relation Types)**：
  * **约束关系**（如 `app_rule_ct_backup_solution`、`data_rule_ct_backup_solution`）：将外部多维度规则指向可用的候选方案。
  * **组装关系**（如 `backup_solution_use_backup_plan`、`backup_solution_use_backup_window`）：连接方案层级与底层策略组件。

### 2.2 备份运行知识网络组成
主要描述备份系统真实运行态中各实体间的数据流转与从属关系，作为运行事实的数据基座：

* **核心实体对象**：
  * `production_system`（生产系统）、`production_resource`（生产资源）、`protected_object`（保护对象）：定义了业务端数据对象的来源层级。
  * `backup_config`（备份配置）、`mysql_backup_config`、`client`（客户端）、`storage_pool`（存储池）、`storage_service`（存储服务）：定义了数据保护系统的各类承载与配置实体。
* **核心关系 (Relation Types)**：
  * `prod_resource_gen_protected_object`：生产资源生成保护对象。
  * `protected_object_has_backup_config`：保护对象与实际备份配置的绑定关系。
  * 配置关联关系（如 `backup_config_to_client`、`backup_config_to_storage_pool` 等）。
* **动作对象 (Action Types)**：
  * 包含 `query_prod_resource_backup_cfg`、`query_protected_object_backup_config` 等，支持在智能对话中动态查询各对象的备份配置与环境拓扑。

## 3. 备份方案推理逻辑

在实际业务场景（如“为生产资源 xxx 推荐备份方案”）中，备份方案的推荐并非仅靠规则引擎独立完成，而是通过“运行事实识别 -> 备份方案推荐”的双网协同模式实现。推理逻辑分为两阶段：

### 阶段一：运行事实识别与特征提取
本阶段由**备份运行知识网络**主导，为后续规则推荐提供坚实的客观上下文数据支撑：

* **步骤 1：意图与实体识别**
  * 解析用户请求意图（如：“为名为 `crm-db-01` 的 MySQL 生产资源推荐备份方案”）。
  * 提取关键实体标识（如 `production_resource` 名称、`protected_object` 名称或 ID 等）作为事实查询入口。

* **步骤 2：运行环境特征提取与拓扑关联**
  * 触发底层 Action（如 `query_prod_resource_backup_cfg`），获取该生产资源的真实运行画像。
  * 利用知识网络拓扑（如 `prod_system_has_prod_resource`、`protected_object_to_prod_system`）向上追溯其所属业务线或租户类型。
  * 提取并整合关键特征：真实的应用类型（如 `mysql`）、数据量大小、系统架构、当前是否已存在保护对象配置及可用资源状态（如存储池剩余容量）。

* **步骤 3：事实特征转化为推荐上下文**
  * 将提取到的客观事实数据自动映射为**备份规则知识网络**可识别的上下文参数（如：将提取到的应用类型转化为 `app_object_type=mysql`，将数据容量转化为 `data_size_gb` 条件输入）。
  * 结合用户在对话中补充的期望（如行业类型、目标 RPO），构建出完整的推荐输入上下文。

### 阶段二：备份方案规则推荐与校验
本阶段由**备份规则知识网络**接管，接收上述上下文并执行严密的流式推理，最终输出格式化方案：

* **步骤 4：上下文输入与默认值补齐**
  * 接收完整的输入上下文。
  * 根据“行业类型”匹配 `recommendation_policy`（候选推荐策略），补齐缺失参数（如 `default_rpo_minutes`、`default_backup_window`）并获取评估权重。

* **步骤 5：规则匹配与方案缩圈（初筛）**
  * 基于上下文参数，独立匹配 `app_rule`、`data_rule`、`legal_rule`、`rpo_rule` 等规则对象。
  * 通过网络中的约束关系（如 `app_rule_ct_backup_solution`），逆向追溯对应的 `backup_solution` 候选集合，对各维度圈定的方案集取交集，获得满足刚性约束的候选列表。

* **步骤 6：方案核心组件组装与配置动态汇总**
  * 利用组装关系向下挂载每个候选方案的备份窗口、备份计划与保留计划。
  * 根据附加条件（如 `condition: data_size_gb >= 500G`）动态匹配 `basic_config_rule` 和 `app_config_rule`。
  * 按 `config_category`（配置类别）对命中的配置进行去重和汇总，确保同一类别配置无冲突。

* **步骤 7：综合评分与防冲突校验**
  * 应用推荐策略权重因子（合规、RPO、窗口等）对方案进行综合打分，并按最大候选数量截取最优方案。
  * **事实防冲突校验**：将推荐结果与阶段一提取到的运行环境事实进行比对，提示当前推荐方案与存量配置的差异，或过滤掉不满足客观环境（如可用存储池容量不足）的无效方案。

* **步骤 8：输出层级灵活格式化**
  * 匹配 `app_output_rule`（应用输出规则），针对不同的应用（如 MySQL、Oracle、File）提取专属的视图展示层级名称（如 `solution_section_name`）。
  * 执行输出控制策略，组合出高度适配前端展现且结构清晰的推荐结果，返回给用户。

## 4. 运行事实查询推理逻辑

备份运行知识网络的核心职责是提供真实、客观的环境数据支撑。当用户意图并非“推荐”，而是“纯查询”（如“排查某保护对象的备份失败原因”）时，其推理链路围绕“实体识别 -> 事实获取 -> 拓扑关联 -> 状态呈现”展开：

### 步骤 1：意图与实体识别
* 解析用户查询意图，提取关键实体标识（如 `production_system` 名称或生产资源 ID 等）作为事实查询的入口起点。

### 步骤 2：运行事实获取 (Action 触发)
* 根据识别到的核心实体，触发系统预置的 Action（如 `query_protected_object_backup_config` 等）。
* 从底层保护系统中实时获取保护对象及其直接绑定的备份配置（`backup_config`）事实数据。

### 步骤 3：环境拓扑关联与追溯
* 基于获取到的基础事实数据，利用知识网络中的关系在运行态拓扑中进行多向延展与追溯。
* **向上追溯业务归属**：通过 `protected_object_to_prod_system` 确认当前保护对象属于哪个生产系统。
* **向下延展资源链路**：通过 `backup_config_to_client`、`backup_config_to_storage_pool` 等关系，完整勾勒出数据流向与资源消耗链路。

### 步骤 4：事实组装与客观呈现
* 将跨越多个层级的离散运行态数据进行结构化拼装，形成客观的“运行状态全景图”。该呈现仅反映“当前正在发生什么”，为排障分析、合规审计提供坚实可信的数据基座。
